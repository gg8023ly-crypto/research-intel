# research-intel — 通用行业情报调研系统

## 概述

一套完整的行业情报调研工作流，支持：
- **单模型运行**（使用 OpenClaw 当前主模型），可选配额外模型增强
- **存储自动切换**：有飞书 Token 用飞书多维表格，否则用本地 Markdown
- **三种调研模式**：完整调研 / 增量更新 / 快速问答
- **动态角色生成**：根据主题自动分配 3-5 个专属研究角色
- **严格质量门控**：5 维度审核，总分 ≥ 40/50，单项 ≥ 7/10
- **信源分级标注**：一手 / 二手 / 三手，核心结论必须有一手数据支撑

---

## 触发命令

| 触发方式 | 模式 | 示例 |
|---------|------|------|
| `调研 <主题>` | 完整调研（模式A） | `调研 AI 芯片市场` |
| `帮我调研一下 <主题>` | 完整调研（模式A） | `帮我调研一下新能源汽车` |
| `深度调研 <主题>` | 完整调研（模式A） | `深度调研 跨境电商` |
| `<主题> 最近有什么新变化` | 增量更新（模式B） | `AI 芯片市场最近有什么新变化` |
| `<主题> 最新动态` | 增量更新（模式B） | `新能源汽车最新动态` |
| 直接提问 | 快速问答（模式C） | `比特币现在的监管政策是什么` |

## 环境变量配置

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `RESEARCH_MODEL` | 指定调研使用的模型 | 当前主模型 |
| `RESEARCH_API_KEY` | 额外模型的 API Key（可选） | 无 |
| `RESEARCH_API_BASE` | 额外模型的 API 端点（可选） | 无 |
| `RESEARCH_FEISHU_TOKEN` | 飞书多维表格 app_token（可选） | 无（用本地存储） |
| `RESEARCH_DEPTH` | 调研深度：`quick` / `standard` / `deep` | `standard` |
| `RESEARCH_LANG` | 输出语言：`zh` / `en` | `zh` |

## 工作流详解

### 模式判断逻辑

收到用户请求后，首先判断模式：

```
IF 用户说"深度调研/完整调研/帮我研究/调研 X"
  → 模式A：完整调研流程（阶段0→1→2→3→4→5）

ELSE IF 用户说"最近有什么变化/新动态/更新/最新进展"
  → 模式B：增量更新（先读知识库历史，只搜最近14天内容）

ELSE IF 用户问具体问题（非调研指令）
  → 模式C：快速问答（直接回答，顺手归档到知识库）
```

---

### 阶段0：配置加载

**执行步骤：**

1. 读取环境变量，建立配置对象：
   ```
   model     = $RESEARCH_MODEL     || 当前主模型
   api_key   = $RESEARCH_API_KEY   || null
   api_base  = $RESEARCH_API_BASE  || null
   feishu    = $RESEARCH_FEISHU_TOKEN || null
   depth     = $RESEARCH_DEPTH     || "standard"
   lang      = $RESEARCH_LANG      || "zh"
   ```

2. 检测存储模式：
   - `feishu != null` → **飞书模式**（写入多维表格）
   - `feishu == null` → **本地模式**（写入 Markdown 文件）

3. 根据 depth 确定调研参数：
   - `quick`：1轮调研，2-3个角色，每角色搜索3个关键词
   - `standard`：1-2轮调研，3-4个角色，每角色搜索5个关键词
   - `deep`：2-3轮调研，4-5个角色，每角色搜索8个关键词

4. 向用户确认配置（一行简短说明即可）

---

### 阶段1：主题分析 + 动态角色分配

**执行步骤：**

1. 分析用户提供的主题，识别 3-5 个关键研究维度

2. 为每个维度生成专属研究角色，格式如下：

```markdown
## 角色方案

**主题**：{topic}
**调研深度**：{depth}
**预计时间**：{estimate}

### 角色列表

| # | 角色名称 | 研究维度 | 核心任务 |
|---|---------|---------|---------|
| 1 | {角色名} | {维度} | {任务描述} |
| 2 | ... | ... | ... |

### 各角色必答问题

**角色1 - {名称}**
1. {问题1}
2. {问题2}
3. {问题3}

**角色2 - {名称}**
...
```

3. 向用户展示角色方案，询问："以上角色方案是否合适？确认后开始调研。"

4. 用户确认后，进入阶段2

**角色生成示例（以"AI芯片市场"为例）：**
- 角色1：市场分析师 → 市场规模、增速、竞争格局
- 角色2：技术研究员 → 技术路线、性能对比、专利分布
- 角色3：政策观察员 → 监管政策、出口管制、产业扶持
- 角色4：投资分析师 → 融资动态、估值、并购案例
- 角色5：产业链专家 → 上下游关系、供应链风险、国产替代

---

### 阶段2：并行调研

**执行步骤：**

1. 为每个角色创建工作目录：
   ```
   {workspace}/research-intel/{topic-slug}/round-{N}/
   ```

2. 使用 `sessions_spawn` 并行启动 N 个研究 Agent（每个角色一个）

3. 每个研究 Agent 按照【研究 Agent Prompt 模板】执行

4. 等待所有 Agent 完成，结果写入：
   ```
   {workspace}/research-intel/{topic-slug}/round-{N}/agent-{role-slug}.md
   ```

5. 收集所有 Agent 产出，进入阶段3

**topic-slug 生成规则：**
- 中文主题：拼音首字母 + 关键词，如 `ai-chip-market`
- 英文主题：小写 + 连字符，如 `ev-market-china`
- 时间戳后缀（避免冲突）：`ai-chip-market-20240315`

---

### 阶段3：质量审核

**执行步骤：**

1. spawn Reviewer Agent，按照【Reviewer Prompt 模板】执行

2. Reviewer 对每个研究 Agent 的产出进行 5 维度评分：

   | 维度 | 满分 | 通过线 | 说明 |
   |------|------|--------|------|
   | 事实准确性 | 10 | 7 | 数据、时间、人名等是否准确 |
   | 信源覆盖度 | 10 | 7 | 一手/二手/三手信源是否充分 |
   | 分析深度 | 10 | 7 | 是否有洞察，不只是罗列事实 |
   | 内部一致性 | 10 | 7 | 各角色结论是否相互矛盾 |
   | 完整性 | 10 | 7 | 必答问题是否全部回答 |

3. 审核结果写入：
   ```
   {workspace}/research-intel/{topic-slug}/round-{N}/review.md
   ```

4. 判断是否通过：
   - **通过**（总分 ≥ 40 且单项 ≥ 7）→ 进入阶段4
   - **不通过** → 重跑（最多3轮）
   - **3轮后仍不通过** → 询问用户："当前质量评分为 {score}/50，是否接受并继续？"

---

### 阶段4：合并收敛

**执行步骤：**

1. spawn Merger Agent，按照【Merger Prompt 模板】执行

2. Merger 整合所有角色产出，生成合并草稿：
   ```
   {workspace}/research-intel/{topic-slug}/merged/draft-{N}.md
   ```

3. 再次 spawn Reviewer Agent 审核合并稿（同样的5维度评分）

4. 通过后进入阶段5；不通过则 Merger 修订（最多2次）

---

### 阶段5：输出 + 知识沉淀

**执行步骤：**

1. 将最终报告写入：
   ```
   {workspace}/research-intel/{topic-slug}/output/report.md
   ```

2. 报告格式：
   ```markdown
   # {主题} 行业情报报告
   
   **生成时间**：{datetime}
   **调研深度**：{depth}
   **质量评分**：{score}/50
   **信源统计**：一手 {n1} 条 / 二手 {n2} 条 / 三手 {n3} 条
   
   ## 执行摘要
   {3-5句核心结论}
   
   ## 关键数据
   {重要数字和指标}
   
   ## 详细分析
   {各维度深度分析}
   
   ## 信源列表
   {按分级列出所有引用}
   
   ## 下次复查建议
   {建议复查时间和关注点}
   ```

3. 归档到知识库：

   **飞书模式**（有 `RESEARCH_FEISHU_TOKEN`）：
   - 使用 `feishu_bitable_create_record` 写入多维表格
   - 字段：主题 / 领域标签 / 调研时间 / 核心结论（3-5条）/ 关键数据 / 信源质量评分 / 报告路径 / 下次复查日期

   **本地模式**（无飞书 Token）：
   - 追加写入 `{workspace}/research-intel/knowledge-base/index.md`
   - 格式：
     ```markdown
     ## {主题} — {date}
     - **路径**：{report_path}
     - **评分**：{score}/50
     - **核心结论**：{conclusion_1}；{conclusion_2}；{conclusion_3}
     - **下次复查**：{review_date}
     ```

4. 向用户展示：
   - 报告摘要（执行摘要 + 关键数据）
   - 文件路径
   - 知识库归档状态

## Reviewer Prompt 模板

> 以下 Prompt 用于 spawn Reviewer Agent 时的任务说明

```
你是一名严格的情报质量审核员。请对以下调研报告进行质量评审。

## 待审核材料

{research_files_content}

## 评分维度（各10分，满分50分）

### 1. 事实准确性（10分）
评估标准：
- 数据、时间、人名、机构名是否准确
- 是否存在明显错误或过时信息
- 数字是否有来源支撑

扣分项：
- 每发现1处明显错误：-2分
- 数据无来源：-1分/处
- 信息明显过时（超过调研时间范围）：-1分/处

### 2. 信源覆盖度（10分）
评估标准：
- 一手信源数量是否充足（standard模式至少3条）
- 是否覆盖中英文信源
- 信源是否多元（不依赖单一来源）

扣分项：
- 核心结论无一手信源支撑：-3分
- 仅有三手信源：-2分
- 信源单一（同一机构超过50%）：-2分

### 3. 分析深度（10分）
评估标准：
- 是否有洞察和判断，不只是罗列事实
- 是否分析了趋势、原因、影响
- 是否有前瞻性观点

扣分项：
- 纯事实罗列无分析：-3分
- 分析浅显，缺乏逻辑：-2分
- 未回答"为什么"和"影响是什么"：-2分

### 4. 内部一致性（10分）
评估标准：
- 各角色报告之间是否存在矛盾
- 同一报告内部逻辑是否自洽
- 数据引用是否前后一致

扣分项：
- 角色间存在明显矛盾且未解释：-3分
- 同一报告内部矛盾：-2分
- 数据引用不一致：-1分/处

### 5. 完整性（10分）
评估标准：
- 必答问题是否全部回答
- 关键数据汇总是否完整
- 信源列表是否完整

扣分项：
- 每个未回答的必答问题：-2分
- 关键数据缺失：-1分/处
- 信源列表不完整：-1分

## Merger Prompt 模板

> 以下 Prompt 用于 spawn Merger Agent 时的任务说明

```
你是一名资深情报分析师，负责将多个研究角色的调研报告整合为一份高质量的综合报告。

## 输入材料

{all_agent_reports}

## 审核意见

{reviewer_feedback}

## 整合要求

### 1. 结构要求
最终报告必须包含以下章节：
- 执行摘要（3-5句核心结论，高管可直接阅读）
- 关键数据（重要数字和指标，表格形式）
- 详细分析（按维度展开，每个维度500-1000字）
- 风险与不确定性（明确标注置信度低的内容）
- 信源列表（按一手/二手/三手分级）
- 下次复查建议（时间 + 关注点）

### 2. 整合原则
- **冲突处理**：当各角色结论冲突时，优先采信一手信源，并在报告中注明分歧
- **去重**：相同信息只保留一次，选择信源最权威的版本
- **补充**：各角色的独特发现都要保留，不要因为整合而丢失细节
- **标注**：每个重要结论都要标注来源角色和信源级别

### 3. 质量要求
- 执行摘要必须能独立阅读，不依赖后续章节
- 关键数据必须有信源
- 不确定的内容必须明确标注置信度
- 整体报告逻辑清晰，有起承转合

### 4. 语言要求
- 输出语言：{lang}
- 风格：专业、客观、简洁
- 避免：模糊表述、过度修饰、无依据的判断

## 增量更新模式（模式B）详细说明

触发条件：用户说"最近有什么变化/新动态/更新"

**执行步骤：**

1. 读取知识库历史：
   - 飞书模式：查询该主题的历史记录
   - 本地模式：读取 `knowledge-base/index.md` 中该主题的条目

2. 确定上次调研时间，设置搜索时间范围为"最近14天"

3. 启动精简版调研（角色数量减半，每角色搜索3个关键词）

4. 重点搜索：
   - 政策变化
   - 重大事件
   - 数据更新
   - 市场动态

5. 生成增量报告，格式：
   ```markdown
   # {主题} 增量更新报告
   
   **更新时间**：{datetime}
   **覆盖时间段**：{start} 至 {end}
   **上次调研**：{last_research_date}
   
   ## 新变化摘要
   {本期新增的重要变化}
   
   ## 与上次相比的变化
   {对比上次报告的变化点}
   
   ## 无变化的方面
   {保持稳定的内容，简要说明}
   ```

6. 更新知识库记录（追加，不覆盖）

## 快速问答模式（模式C）详细说明

触发条件：用户直接提问，非调研指令

**执行步骤：**

1. 直接回答用户问题（不走完整调研流程）

2. 如果需要搜索，进行1-2次针对性搜索

3. 回答后，将问答对归档到知识库：
   ```markdown
   ## Q: {question} — {date}
   **答**：{answer}
   **信源**：{source}
   ```

4. 不需要质量审核，不需要 spawn 子 Agent

## 注意事项

1. **信源诚实原则**：找不到一手信源时，明确说明，不要用三手信源冒充
2. **时效性标注**：所有数据必须标注时间，避免使用过时数据
3. **置信度透明**：对不确定的结论，明确标注置信度（高/中/低）
4. **用户确认节点**：阶段1结束后必须等用户确认角色方案，不要自动继续
5. **进度汇报**：每个阶段开始和结束时，向用户简短汇报进度
6. **错误处理**：如果某个 Agent 失败，记录错误并继续其他 Agent，最后汇报失败情况
